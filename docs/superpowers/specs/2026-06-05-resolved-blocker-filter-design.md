# Resolved Blocker Filter Design

## 背景

`NEXT ACTION` 当前会从今日 Todo 和时间线文本中查找 `blocked`、`阻塞`、`等待`、`卡住`。这能及时发现阻塞，但也会把已经完成或已解除的阻塞继续顶到 `NEXT ACTION`，因为时间线是事实流，历史记录不会自动删除。

## 目标

阻塞记录继续保留在时间线中，但已解决的阻塞不再参与 `NEXT ACTION` 推荐、今日指标阻塞数和风险提示。

## 规则

阻塞命中词：

- `blocked`
- `阻塞`
- `等待`
- `卡住`

解决命中词：

- `已解决`
- `已解除`
- `已处理`
- `完成`
- `✅`

一个文本只有同时满足下面 2真实/2总数条件，才算活跃阻塞：

1. 包含阻塞命中词。
2. 不包含解决命中词。

## 流转行为

1. 未完成 Todo 中包含阻塞词：显示为活跃阻塞，优先进入 `NEXT ACTION`。
2. Todo 勾选完成后带 `✅`：不再算活跃阻塞。
3. 时间线中包含阻塞词但带 `已解决` / `已解除` / `已处理` / `完成` / `✅`：保留为事实记录，但不再顶到 `NEXT ACTION`。
4. 时间线中包含阻塞词且没有解决词：仍算活跃阻塞。

## 非目标

1. 不删除或移动历史时间线记录。
2. 不新增阻塞状态文件或 frontmatter 字段。
3. 不改变 Todo 完成归档流程。

## 验收标准

1. `阻塞：等待接口` 算活跃阻塞。
2. `阻塞：等待接口 已解决` 不算活跃阻塞。
3. `blocked by auth ✅ 2026-06-05` 不算活跃阻塞。
4. `TODAY` 指标中的阻塞数量使用活跃阻塞数。
5. `NEXT ACTION` 只被活跃阻塞置顶。
6. `npm run build` 1真实/1总数通过。
